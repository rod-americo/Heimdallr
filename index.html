Heimdallr

Operational infrastructure for imaging intake, preparation, segmentation, deterministic metrics, and artifact delivery.

Heimdallr receives DICOM studies, prepares selected series, converts to NIfTI, orchestrates TotalSegmentator-backed segmentation, tracks SQLite queue state, and delivers artifacts through dashboards, DICOM egress, and HTTP callbacks.

View repository Read architecture

Purpose

A governed operational layer for radiology image pipelines.

Heimdallr keeps PACS-facing intake, DICOM conversion, segmentation orchestration, deterministic measurements, queue state, runtime artifacts, and delivery workflows in one auditable open-source stack. It is designed for technical operators who need clear state transitions and repeatable outputs, not autonomous clinical decision making.

Why Heimdallr

Built around operational boundaries.

Radiology image processing has infrastructure concerns that should stay separate from reporting, NLP, and proprietary intelligence layers.

01

DICOM-native intake

C-STORE study handoff, ZIP upload ingress, and deterministic study grouping by StudyInstanceUID.

02

Traceable runtime state

Case workspaces, SQLite queues, metadata payloads, and generated artifacts remain explicit and inspectable.

03

Deterministic outputs

Segmentation and metrics jobs produce reproducible infrastructure artifacts for qualified review.

04

Clear repository scope

Report drafting, prompt workflows, and clinical narrative intelligence remain outside this codebase.

Architecture

From incoming studies to delivered artifacts.

  1. 01 Intake

    DICOM C-STORE or ZIP uploads enter the stable spool.

  2. 02 Prepare

    Metadata extraction, selected-series policy, and DICOM-to-NIfTI conversion.

  3. 03 Segment

    TotalSegmentator tasks run from host-local profiles with queue heartbeats.

  4. 04 Measure

    Metrics jobs write results JSON, overlays, PDFs, and DICOM artifacts.

  5. 05 Deliver

    Dashboards, DICOM egress, event dispatch, and final callback packages expose the result.

Core capabilities

Resident services with explicit responsibilities.

Control plane

FastAPI dashboard, upload ingress, patient/results API, and deterministic PDF export.

Prepare worker

ZIP claiming, DICOM scan, metadata extraction, conversion, and segmentation enqueue.

Segmentation worker

Selected-series reuse, TotalSegmentator supervision, canonical NIfTI, and output layout.

Metrics worker

Dynamic job loading, dependency graph execution, generated artifacts, and delivery enqueue.

DICOM egress

Outbound C-STORE delivery through retry-aware queue workers and peer configuration.

Operational guards

Resource telemetry, disk usage protection, Textual TUI, and structural validation scripts.

First steps

Run the stack with a single Python 3.12 environment.

Host-specific values stay outside Git. Create local JSON config from the versioned examples, inject secrets through the host supervisor or container runtime, and keep mutable data under ignored runtime paths.

README Operations API 
git clone git@github.com:rod-americo/Heimdallr.git
cd Heimdallr
python3.12 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

cp config/segmentation_pipeline.example.json config/segmentation_pipeline.json
cp config/metrics_pipeline.example.json config/metrics_pipeline.json

.venv/bin/python -m heimdallr.control_plane
.venv/bin/python -m heimdallr.prepare
.venv/bin/python -m heimdallr.segmentation
.venv/bin/python -m heimdallr.metrics
